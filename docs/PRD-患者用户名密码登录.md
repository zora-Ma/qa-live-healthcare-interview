## 1. 产品概述

### 1.1 产品背景

**现状分析**:
当前 Healthcare QA 系统的患者身份验证方式存在以下问题：

1. **验证方式不够安全**: 目前患者只需输入“姓名+生日”即可登录，这些信息容易被他人获取
2. **缺乏唯一标识**: 患者数据存储在 patient-user.json 中，没有用户名和密码字段
3. **无法追溯历史**: 多个患者可能使用相同的姓名和生日，导致身份混淆
4. **无法修改凭证**: 患者无法修改自己的登录信息
5. **单一登录方式**: 仅支持一种验证方式，无法满足所有用户的使用习惯

**业务需求**:
- 在保留原有“姓名+生日”登录方式的基础上，增加“用户名+密码”登录方式
- 提供安全的身份验证机制，支持密码加密存储
- 支持患者注册、登录、修改密码等完整功能
- 保证患者数据隐私和安全
- 与现有系统无缝集成，不影响医生端功能及原有患者的登录体验

### 1.2 产品目标

| 目标编号 | 目标描述 | 衡量指标 |
|---------|---------|---------|
| GOAL-001 | 提供安全的患者登录方式 | 使用 BCrypt 加密存储密码，通过安全审计 |
| GOAL-002 | 简化患者注册流程 | 注册步骤 ≤ 3 步，注册成功率 ≥ 95% |
| GOAL-003 | 支持密码管理功能 | 密码修改成功率 ≥ 99% |
| GOAL-004 | 保持向后兼容 | 现有患者数据 100% 可用，零数据丢失 |
| GOAL-005 | 提升用户体验 | 支持两种登录方式自由切换，满足不同用户需求 |

### 1.3 产品范围

**包含范围 (In Scope)**:
- 患者用户名注册
- 患者用户名密码登录（新增）
- 保留原有的“姓名+生日”登录方式
- 默认密码自动生成（手机号后四位）
- 修改密码功能
- 患者数据持久化（数据库存储）
- 患者认证 API 开发
- 前端登录页面改造（支持双模式切换）

**不包含范围 (Out of Scope)**:
- 患者手机号验证（本期不涉及短信验证）
- 患者邮箱找回（本期不涉及邮箱功能）
- 第三方登录（微信、支付宝等）
- 患者实名认证

### 3.2 功能详细描述

#### F-PAT-001: 患者注册

**功能描述**:
> 患者通过填写基本信息（姓名、手机号）注册账号，系统自动生成用户名和默认密码。

**用户故事**:
```
作为 新患者
我想要 注册一个账号
以便于 安全地登录系统进行问诊
```

**验收标准**:
- [ ] 注册页面包含：姓名、手机号、性别（可选）
- [ ] 手机号格式验证（中国大陆 11 位）
- [ ] 系统自动生成用户名（格式：手机号）
- [ ] 系统自动生成默认密码（手机号后四位）
- [ ] 注册成功后显示用户名和默认密码提示
- [ ] 注册成功后自动跳转到登录页面
- [ ] 重复注册（相同手机号）给出提示

**业务规则**:
1. 用户名生成规则：使用手机号，如 13800001234
2. 默认密码规则：手机号后四位，如 1234
3. 手机号唯一性：同一手机号只能注册一次
4. 数据持久化：注册成功后数据写入数据库 patients 表

**接口依赖**:
- POST /api/patients/register

**UI/UX 要求**:
- 页面路径: /patient/register
- 导航入口: 问诊页面新增"注册"按钮
- 表单验证: 实时验证手机号格式

**优先级**: P0（必须实现）

#### F-PAT-002: 患者登录

**功能描述**:
> 患者通过用户名和密码登录系统，进入问诊页面。

**用户故事**:
```
作为 已注册患者
我想要 使用用户名和密码登录
以便于 访问我的问诊记录并提交新问题
```

**验收标准**:
- [ ] 登录页面包含：用户名、密码输入框
- [ ] 登录成功跳转到问诊页面（/consultation）
- [ ] 登录失败显示错误提示（用户名或密码错误）
- [ ] 密码输入框支持显示/隐藏密码
- [ ] 登录失败超过 5 次锁定账号 15 分钟

**业务规则**:
1. 用户名可以是手机号
2. 密码验证使用 BCrypt 解密比对
3. 登录成功后返回患者完整信息
4. 会话有效期 24 小时

**接口依赖**:
- POST /api/patients/login

**UI/UX 要求**:
- 页面路径: /patient/login
- 登录页面提供“账号密码登录”和“姓名生日登录”两种模式的切换选项
- 默认显示“账号密码登录”，提供“忘记密码”入口（本期仅提示联系客服）
- “姓名生日登录”模式保持原有交互逻辑不变

**优先级**: P0（必须实现）

#### F-PAT-003: 修改密码

**功能描述**:
> 已登录患者可以修改自己的登录密码。

**用户故事**:
```
作为 已登录患者
我想要 修改我的密码
以便于 提高账号安全性
```

**验收标准**:
- [ ] 修改密码页面包含：原密码、新密码、确认新密码
- [ ] 原密码验证通过后才能修改
- [ ] 新密码强度验证（至少 6 位，包含字母和数字）
- [ ] 新密码和确认密码必须一致

**业务规则**:

2. 新密码长度 6-20 位
3. 密码强度要求：至少包含字母和数字
4. 修改成功后原会话失效，需重新登录

**接口依赖**:
- POST /api/patients/change-password

**优先级**: P0（必须实现）

#### F-PAT-004: 默认密码生成

**功能描述**:
> 系统为新注册患者自动生成默认密码（手机号后四位）。

**用户故事**:
```
作为 系统
我想要 为患者生成默认密码
以便于 患者首次登录时有可用的凭证
```

**验收标准**:
- [ ] 注册时自动提取手机号后四位
- [ ] 密码使用 BCrypt 加密后存储
- [ ] 注册成功后向用户展示默认密码（仅展示一次）
- [ ] 默认密码符合密码强度要求（4 位数字）

**业务规则**:
1. 默认密码为手机号后四位（如 1234）
2. 必须使用 BCrypt 加密存储
3. 首次登录后可修改密码

**实现方式**:
```java
// 示例代码
String phone = "13800001234";
String defaultPassword = phone.substring(phone.length() - 4); // "1234"
String encryptedPassword = BCrypt.hashpw(defaultPassword, BCrypt.gensalt());
```

**优先级**: P0（必须实现）

#### F-PAT-008: 历史数据迁移

**功能描述**:
> 将现有的 patient-user.json 数据导入数据库，并生成默认用户名和密码。

**用户故事**:
```
作为 系统管理员
我想要 将现有患者数据迁移到数据库
以便于 保证历史数据的可用性和连续性
```

**验收标准**:
- [ ] 所有现有患者数据成功导入数据库
- [ ] 为每个患者生成默认用户名（手机号）
- [ ] 为每个患者生成默认密码（手机号后四位）
- [ ] 迁移后前端功能不受影响
- [ ] 提供迁移日志和回滚方案

**迁移策略**:

1. **数据源分析**: 
   - 原始文件: `web/qa-web/src/data/patient-user.json`
   - 数据特征: 包含 5 条测试数据，手机号已脱敏 (如 `138****1234`)。

2. **数据映射与清洗规则**:
   - **ID 映射**: 保持原有 ID 不变 (如 `patient001`)。
   - **手机号还原**: 由于原始数据脱敏，初始化时采用预设的完整手机号 (见下方数据清单)。
   - **用户名生成**: 直接映射为完整的 11 位手机号。
   - **密码生成**: 提取手机号后四位，经 BCrypt 加密后存入 `password` 字段。

3. **初始化数据清单 (SQL 示例)**:
```sql
-- 假设 BCrypt 加密后的 '1234' 为: $2a$10$N9qo8uLOickgx2ZMRZoMy.MrEo6Jz3Xj/9i6K7y8k5l6m7n8o9p0q
INSERT INTO patients (id, username, password, name, phone, birthday, gender) VALUES
('patient001', '13800001234', '$2a$10$...', '赵明', '13800001234', '1985-03-15', '男'),
('patient002', '13900005678', '$2a$10$...', '孙丽', '13900005678', '1990-07-22', '女'),
('patient003', '13700009012', '$2a$10$...', '周杰', '13700009012', '1978-11-08', '男'),
('patient004', '13600003456', '$2a$10$...', '吴芳', '13600003456', '1995-05-20', '女'),
('patient005', '13500007890', '$2a$10$...', '郑浩', '13500007890', '1988-09-12', '男');
```

4. **执行方式**:
   - **手动执行**: 开发人员或运维人员直接连接 MySQL 数据库（如通过 phpMyAdmin 或命令行），执行 `02-patient-data.sql` 脚本。
   - **脚本位置**: `doc/docker/mysql/init/02-patient-data.sql`。
   - **注意事项**: 确保在执行前已设置字符集 `SET NAMES utf8mb4;`，且 `patients` 表已创建。

5. **回滚方案**:
   - 初始化脚本需具备幂等性（使用 `INSERT IGNORE` 或先 `DROP TABLE IF EXISTS`）。
   - 保留原始 `patient-user.json` 作为数据备份。

**优先级**: P0（必须实现）

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 要求 | 测试方法 |

## 文档编写规范

### 必填项

以下章节为必填项，不可省略：

1. ✅ 文档元信息
2. ✅ 产品概述（背景、目标、范围）
3. ✅ 用户角色与权限
4. ✅ 功能需求（至少包含功能清单和一个功能的详细描述）
5. ✅ 非功能需求（至少包含性能需求）
6. ✅ 接口规范（至少包含接口清单和一个接口的详细定义）
7. ✅ 测试策略（至少包含测试范围和测试用例）

### 选填项

以下章节可根据实际需求省略或简化：

- 📝 数据模型（如不涉及复杂数据关系可简化）
- 📝 风险管理（小型项目可简化）
- 📝 实施计划（可根据项目管理方式调整）

### 格式规范

1. **优先级标注**:
   - P0: 必须实现（MVP 功能）
   - P1: 应该实现（重要功能）
   - P2: 可以实现（锦上添花）

2. **用户故事格式**:
   ```
   作为 <角色>
   我想要 <动作>
   以便于 <价值>
   ```

3. **验收标准格式**:
   - 使用复选框列表 [ ]
   - 每条标准应该是可验证的

4. **接口文档格式**:
   - 包含请求方法、路径、参数、响应示例
   - 说明所有可能的状态码
